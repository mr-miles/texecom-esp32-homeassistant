<task type="auto">
  <name>Task 2: Byte-integrity and session stability stress test</name>
  <files>.planning/phases/01-serial-to-tcp-bridge/validation-evidence.md, .planning/phases/01-serial-to-tcp-bridge/images/</files>
  <action>
Two separate stress tests, evidence appended to the validation document.

**Byte-integrity test**:
- Attach a passive serial tap (FTDI cable in listen-only mode on the Atom-to-panel TX line, or a second logic analyser channel) so we can count raw bytes independently of ESPHome's reporting.
- Run Wintex "Receive" on a panel with enough programmed content to generate a large download (ideally 30+ seconds of sustained traffic).
- Count bytes seen by: (a) the passive tap, (b) the ESPHome log counter, (c) Wintex's own status bar if visible.
- Pass criterion: the three numbers match exactly (or within ±1 byte, which can be attributed to start/stop timing at capture boundaries).
- Include a screenshot of the passive tap capture tool showing the final byte count and the ESPHome log showing its matching count.

**Session-stability test**:
- Disconnect Wintex and reconnect 5 times in a row. Each cycle: close Wintex UDL, wait 5 seconds, reconnect, perform a short read, close.
- Pass criterion: all 5 reconnects succeed; no ESPHome crashes; `is_bridge_mode()` returns true when connected and false when disconnected (observable in logs).
- Include a single ESPHome log excerpt showing all 5 Monitor→Bridge→Monitor transitions.

**Concurrent-connection test**:
- While Wintex is actively connected and doing a read, attempt to telnet to port 10001 from a second machine (`nc <atom-ip> 10001` or `telnet <atom-ip> 10001`).
- Pass criterion: the second connection is rejected immediately; the Wintex read continues uninterrupted and completes successfully.
- Capture both the rejected-connection terminal output and the ESPHome WARN log line noting the rejection.

Append each test as its own `## Test 2.X: ...` section with screenshots, log excerpts, and pass/fail.
