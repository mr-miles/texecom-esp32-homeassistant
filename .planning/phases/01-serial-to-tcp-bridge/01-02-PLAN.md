<task type="auto">
  <name>Task 2: Implement the TCP↔UART bridge</name>
  <files>components/texecom/texecom.cpp, components/texecom/texecom.h</files>
  <action>
Fill in the core byte-pumping logic inside the `Texecom` component. Two ring buffers (UART→TCP and TCP→UART), each 1024 bytes minimum. Backpressure strategy:
- If UART→TCP ring fills because the client is slow, drop oldest bytes and log a warning (Wintex re-sends on timeout — dropping is recoverable).
- If TCP→UART ring fills, DO NOT drop — pause reading from the TCP socket until the buffer drains. Panel UART is the bottleneck; losing config-write bytes would corrupt the panel.

Use ESPHome's `socket` helpers (ESPHome 2023.x+ has a portable socket API). Non-blocking accept + non-blocking recv + non-blocking send. No threads — everything runs in ESPHome's cooperative `loop()`.

Expose `bool is_bridge_mode() const` as a public accessor. Log a single INFO-level line when transitioning between Monitor and Bridge modes, with the remote client's IP.

Graceful disconnect handling:
- On `recv()` returning 0: close the socket, return to Monitor mode, flush outbound UART buffer first (give the panel a chance to finish its in-flight response).
- On socket errors other than `EAGAIN`/`EWOULDBLOCK`: same as above but log at WARN.
- On component `setup()` restart: clear both ring buffers.

Second-client rejection:
- `accept()` while a client is active → accept the new socket just to close it (with a best-effort "busy" sentinel byte written first if feasible), then continue.
- Log at WARN including the rejected IP.

Sizeable implementation — allow up to 300 lines in `texecom.cpp`. Keep public API minimal; internal helpers can be file-local functions or private methods.
