<task type="auto">
  <name>Task 1: Select and document the UART wiring</name>
  <files>.planning/hardware/phase-1-wiring.md</files>
  <action>
Identify the Premier 24's COM1 (PC interface) pin header pinout by reading the installation manual. Record TX, RX, GND, and any +V pins with voltage levels.

**GPIO assignment is pre-locked**: use **GPIO5 for TX (Atom S3 → panel)** and **GPIO6 for RX (panel → Atom S3)**. These are the top-header pins on the Atom S3 (immediately accessible, away from boot/strapping pins, away from the onboard USB-CDC console on GPIO19/20). Document this choice in the wiring doc as the spec — Plan 02 hard-codes these pins. If a physical conflict forces a change, update both this doc and `esphome/texecom-bridge.yaml` together.

Determine whether the panel's 5V TTL output on its TX line needs protection before reaching the Atom S3's 3.3V GPIO input. Options, in order of preference:
1. Direct connection if the Atom S3 input is documented as 5V-tolerant on that pin (check the ESP32-S3 datasheet table — most GPIOs are NOT 5V-tolerant).
2. Resistor divider (e.g. 2k2 / 3k3 or 10k / 15k) on the panel-TX → Atom-RX line.
3. Defer to a proper level shifter in Phase 4 — flag this decision here.

Note that the Atom S3 → panel direction is usually fine at 3.3V into a 5V TTL input, since most CMOS/TTL inputs register anything above ~2.0V as logic high. Confirm from the panel spec if possible.

Create `.planning/hardware/phase-1-wiring.md` with sections:
- **Hardware**: Atom S3 part number, Premier 24 panel revision (if visible), cable type used for the test.
- **Panel COM1 pinout**: table of pins with function + voltage level.
- **Atom S3 GPIO assignments**: table of pins with function, justification for choice.
- **Wiring diagram**: ASCII or text description of every connection (Panel pin X → protection component → Atom S3 pin Y).
- **Baud rate**: **19200 8N2** (8 data bits, no parity, **2 stop bits** — confirmed by the RoganDawes ESPHome_Wintex reference for the Premier Wintex UART; also matches the universaldiscoverymethodology.com 2015 article which states "19200 baud, 8 data bits, no parity, 2 stop bits"). Do NOT use 8N1 — Wintex framing depends on the second stop bit.
- **Protection components**: if resistor dividers used, specify exact values and wattage.
- **Power**: "USB-C from PC or 5V wall wart during Phase 1-3; panel power integration is Phase 4."
- **Photos placeholder**: reserve a section for photos to be added once wired.

Length target: 40-80 lines. Keep it usable as a permanent reference — this document is the single source of truth for wiring until Phase 4 rewrites it.

  <name>Task 2: Write the ESPHome bring-up YAML</name>
  <files>esphome/bringup-uart-logger.yaml</files>
  <action>
Create `esphome/bringup-uart-logger.yaml` — a minimal ESPHome configuration whose sole job is to flash onto the Atom S3, connect to Wi-Fi, and log every byte received on the panel UART at the chosen baud rate. No custom component yet; this is a sanity check using only built-in ESPHome features.

Required sections:
- `esphome:` — name: `texecom-bringup`, friendly name: `Texecom Bring-up`
- `esp32:` — board: `m5stack-atoms3` (or `esp32-s3-devkitc-1` if that's not recognised), framework: `arduino` (ESPHome's default for most external-component work; we'll revisit in Plan 02)
- `wifi:` — use `!secret wifi_ssid` / `!secret wifi_password`; enable `ap:` fallback for first-boot provisioning
- `captive_portal:`
- `logger:` — baud_rate: 115200 (USB-CDC log, NOT the panel UART), level: DEBUG
- `api:` — for ESPHome dashboard integration
- `ota:` — password: `!secret ota_password`
- `uart:` — id: `panel_uart`, tx_pin: GPIO5, rx_pin: GPIO6, **baud_rate: 19200**, data_bits: 8, parity: NONE, **stop_bits: 2**, debug: { direction: BOTH, dummy_receiver: true, after: { bytes: 16, timeout: 100ms }, sequence: [ { lambda: "ESP_LOGD(\"uart\", \"direction=%s bytes=%s\", (direction == UART_DIRECTION_RX ? \"RX\" : \"TX\"), format_hex_pretty(bytes).c_str());" } ] }

Add a `secrets.yaml.example` alongside with placeholder entries for `wifi_ssid`, `wifi_password`, `ota_password`, and `api_encryption_key` (32 bytes, base64) — point users to generate the api key via `esphome dashboard` or `openssl rand -base64 32`.

Also add an `.gitignore` entry pattern for `secrets.yaml` so real credentials never get committed.

Length target: 20-40 lines for the YAML. Comments explaining any non-obvious choices are welcome.
