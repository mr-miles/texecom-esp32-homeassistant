Add an on-device capture layer to the `texecom` ESPHome component. Every byte the bridge moves in either direction is timestamped and recorded into rotating LittleFS files, downloadable over HTTP. This is the data-collection foundation Plan 02-02 needs to reverse-engineer the Wintex protocol — without it, the grammar work is hallucination.

Purpose: Turn the user's bench time (running real Wintex against the device) into reusable capture artefacts.
Output: `components/texecom/capture.{h,cpp}`, integration into `texecom.cpp`, YAML option, host-side tests.
</objective>

<task type="auto">
  <name>Task 1: Capture class — buffer, framing, serialization</name>
  <files>components/texecom/capture.h, components/texecom/capture.cpp</files>
  <action>
Create a `Capture` class that consumes timestamped byte events and produces a structured binary format.

**Event record**:
- direction: 1 byte (0=panel→client, 1=client→panel)
- timestamp_us: 8 bytes little-endian (relative to capture start, not wall-clock — wall-clock unavailable on first boot)
- length: 2 bytes little-endian
- bytes: variable

**File header** (once per file):
- magic: 4 bytes — `"TXCP"` (Texecom Capture)
- version: 2 bytes (= 1)
- start_unix_s: 4 bytes — best-effort wall-clock if SNTP synced, else 0
- panel_model_name: 16 bytes null-padded ASCII (`"Premier 24"` for now)
- baud_rate: 4 bytes (= 19200)
- reserved: 6 bytes zero — keep header at 32 bytes total

**Class API** (`capture.h`):
```cpp
namespace esphome {
namespace texecom {

class Capture {
 public:
  enum class Mode : uint8_t { None = 0, BridgeOnly = 1, MonitorOnly = 2, Both = 3 };

  void set_mode(Mode m);
  void set_panel_name(const std::string &name);
  void set_max_file_bytes(size_t bytes);
  void set_root_path(const std::string &path);  // e.g. "/captures"

  void on_session_start();   // emit fresh file header, reset clock
  void on_session_end();     // close current file, flush
  void record(uint8_t direction, const uint8_t *bytes, size_t len);

  // Optional non-LittleFS sink for unit tests:
  void set_sink_for_test(std::function<void(const uint8_t*, size_t)> sink);

 private:
  // serialize_event_, serialize_header_ — pure functions (testable)
  // flush_, rotate_ — LittleFS-touching (on-device only)
};

}  // namespace texecom
}  // namespace esphome
```

The serialization helpers (`serialize_event`, `serialize_header`) MUST be:
- Free functions or `static` methods (not requiring a Capture instance)
- Pure: byte-in, byte-out, no I/O
- Header-visible so unit tests can call them directly

The LittleFS-touching code goes in `capture.cpp` behind `#ifdef USE_ARDUINO` (same pattern as the AsyncTCP code in `texecom.cpp`) so host tests build without it.

**Rotation**:
- New file when `current_file_bytes_ >= max_file_bytes_` (default 256 KB, configurable)
- New file when `on_session_end()` is called (Bridge mode disconnects)
- New file when `on_session_start()` is called (Bridge mode begins)
- Filename: `{root_path}/wintex-{unix_ts or boot_ms}.bin` — use unix_ts if SNTP available, else boot_ms with a leading zero-padded counter to avoid collisions

**Hex-dump sidecar**:
- For each `.bin`, write a `.txt` with `xxd`-style output (16 bytes/line, offset, hex, ASCII)
- Lazily generated when downloaded, OR written incrementally — either is fine; pick whichever is simpler. Document the choice in the SUMMARY.

**Buffering**:
- In-memory ring buffer of N events (default 64) so individual `record()` calls don't pay the LittleFS cost. Flush triggered by buffer-full, by 100 ms inactivity (timer), and by `on_session_end()`.

**Wiring in `texecom.cpp`**:
- `Texecom` owns a `Capture capture_;` member
- In `on_new_client_` (Bridge entry): `capture_.on_session_start()`
- In `on_client_disconnect_`: `capture_.on_session_end()`
- In `pump_uart_to_tcp_` after reading from UART: `capture_.record(0, bytes, len)` (when Bridge mode active OR Monitor mode and Mode is `MonitorOnly`/`Both`)
- In `on_client_data_` after pushing to ring buffer: `capture_.record(1, bytes, len)` (when Bridge active)
- In `setup()`: configure capture from YAML-supplied options (panel name from `model_->name()`, mode from option, max_file_bytes from option, root_path default `/captures`)

Important: the byte-recording calls must NOT re-block or fail-loop the bridge. If the capture buffer is full and can't flush, log a single WARN and drop the record (the bridge keeps moving bytes — capture loss is preferable to bridge stall). Track dropped-event count in `dump_config()`.

```python
CONF_RECORD_WHEN = "record_when"

}
```
Default `record_when: bridge`, `capture_max_file_bytes: 262144`, `capture_root: "/captures"`.

<task type="auto">
  <name>Task 3: Host-side unit tests for the capture format</name>
  <files>tests/CMakeLists.txt, tests/test_capture.cpp</files>
  <action>
Add `tests/test_capture.cpp` to the CMake target. At least 6 TEST_CASEs:

1. **Header round-trip** — `serialize_header({panel="Premier 24", baud=19200, start_unix=1717000000})` → bytes → parse back → assert all fields recovered. Header is exactly 32 bytes.
2. **Single event round-trip** — `serialize_event(dir=0, ts_us=12345, bytes={0x01,0x02,0x03})` → bytes → parse → assert direction, timestamp, length, payload all recovered.
3. **Multiple events back-to-back** — write 5 events sequentially, parse the resulting buffer linearly, assert all 5 recovered in order.
4. **Magic bytes** — header starts with `'T','X','C','P'`. Test fails loudly if anyone changes the magic.
5. **Length encoding** — write an event with payload length 0, 1, 256, 1023; assert little-endian 2-byte length round-trips correctly.
6. **Direction value** — accept only 0 or 1; values outside that range are caller error (you may assert/abort or return false — doc the choice).
