**`tests/test_wintex_decoder.cpp`** — at least 8 TEST_CASEs, each:
1. Loads the fixture bytes
2. Feeds them byte-by-byte into a `WintexDecoder`
3. Asserts exactly one event is emitted
4. Asserts the event variant matches the expected type and field values

Plus negative-path tests:
- **Bad checksum**: take a known-good fixture, flip a byte, confirm decoder increments `bad_checksum_count()` and emits no event
- **Unknown message ID**: synthesise a frame with a message ID not in the PanelModel table; confirm `unknown_message_count()` increments and no event emitted
- **Partial frame**: feed half the bytes of a fixture, then assert no event yet; feed the rest, assert one event

**Fixture loader helper** (in the test file, not the component):
```cpp
static std::vector<uint8_t> load_fixture(const std::string &name) {
  std::ifstream f("tests/fixtures/" + name, std::ios::binary);
  REQUIRE(f.is_open());
  return std::vector<uint8_t>(std::istreambuf_iterator<char>(f), {});
}
```
Note: ctest runs from the build directory by default; use `set_tests_properties(... PROPERTIES WORKING_DIRECTORY ${CMAKE_SOURCE_DIR})` in CMakeLists or pass the fixture root via a compile-time `-D` macro. Pick whichever is cleaner; document the choice.

Update `tests/CMakeLists.txt` to include `test_wintex_decoder.cpp` and copy or reference the fixtures appropriately.

  <files>components/texecom/texecom.h, components/texecom/texecom.cpp, esphome/texecom-bridge.yaml</files>
  <action>
**`Texecom` class additions**:
- New member: `WintexDecoder decoder_;` (constructed with `model_` pointer in `setup()`)
- Modify `pump_uart_to_tcp_`: in addition to feeding the ring buffer, when `!is_bridge_mode()`, feed each byte to `decoder_.on_byte(b)`
- After feeding bytes, in the same loop tick, drain decoder events: `while (decoder_.has_event()) { auto evt = decoder_.pop_event(); on_decoded_event_(evt); }`
- New private method `on_decoded_event_(const WintexEvent &evt)` — for Phase 2 it just `ESP_LOGD(TAG, "...")` per-variant; Phase 3 will replace this with MQTT publish calls
- In `dump_config()`: report decoder counters (bad checksums, unknown messages, events emitted)

**Bridge-mode behaviour**:
- When entering Bridge mode (`on_new_client_`): call `decoder_.reset()` (add this method) so a fresh stream starts when Bridge ends
- When in Bridge mode: do NOT feed the decoder. Bytes still flow through the bridge byte-for-byte; capture (from Plan 02-01) still records them; decoder is silent.

**Optional YAML option**: `decode_in_bridge_mode: false` (default) — leave as a future-proofing option but don't implement true behaviour now (PROJECT.md decision is Wintex-exclusive, MQTT pauses).

**README update**:
Add a "Phase 2: Decoded events" section to the repo README briefly describing what events appear in the device log when in Monitor mode.
