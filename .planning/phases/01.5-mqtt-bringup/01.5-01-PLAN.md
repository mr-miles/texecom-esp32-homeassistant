```yaml
mqtt:
  broker: !secret mqtt_broker_host
  username: !secret mqtt_broker_username
  password: !secret mqtt_broker_password
  # Default port 1883; bump only if the user runs Mosquitto elsewhere.
  # Discovery is enabled by default — entities auto-appear in HA.

  birth_message:
    topic: texecom-bridge/status
    payload: online
```

Add three sensor declarations to `esphome/texecom-bridge.yaml`. Place them after the `texecom:` block at the bottom of the file.

**Connectivity binary_sensor** (driven by the LWT/birth status topic):

```yaml
binary_sensor:
  - platform: template
    name: "Texecom Bridge Status"
    id: bridge_status
    device_class: connectivity
    # ESPHome's mqtt component automatically publishes the device's
    # availability via the topic_prefix/status pattern. The HA-side
    # discovery message ties this entity to the same availability
    # topic, so the entity goes "unavailable" when the will message
    # fires.
    lambda: |-
      return true;
```

Note: ESPHome's MQTT component already publishes a per-device availability topic that HA discovery uses to drive entity availability state. The `binary_sensor.template` above gives HA a concrete on/off entity (`Texecom Bridge Status`) that surfaces the same signal as a first-class user-visible boolean. If review surfaces that ESPHome's built-in MQTT availability is already sufficient and a separate `binary_sensor` is redundant, the senior-developer may instead use a `binary_sensor.connectivity` platform pointed at the LWT topic, or rely on the auto-published availability — document the choice in the SUMMARY.

**CPU temperature** (ESP32-S3 internal):

```yaml
  - platform: internal_temperature
    name: "Texecom Bridge CPU Temperature"
    id: bridge_cpu_temp
    update_interval: 60s
```

Wait — `internal_temperature` is a `sensor` not a `binary_sensor`. Move it into the `sensor:` block:

```yaml
sensor:
  - platform: internal_temperature
    name: "Texecom Bridge CPU Temperature"
    id: bridge_cpu_temp
    update_interval: 60s

  - platform: wifi_signal
    name: "Texecom Bridge WiFi Signal"
    id: bridge_wifi_signal
    update_interval: 60s
```

Each sensor's `name:` becomes the HA entity friendly name; ESPHome derives the entity_id from the device name + sensor name (so `sensor.texecom_bridge_cpu_temperature`, `sensor.texecom_bridge_wifi_signal`, `binary_sensor.texecom_bridge_status`).

Make sure `update_interval: 60s` matches the success criterion ("≤60s for telemetry").

**Write `.planning/phases/01.5-mqtt-bringup/01.5-VALIDATION.md`** as a step-by-step the user follows once hardware is ready. Cover at least:

1. **Pre-flight**: ensure `esphome/secrets.yaml` has real values for `mqtt_broker_host`, `mqtt_broker_username`, `mqtt_broker_password`. Note that the broker must already be reachable from the device's LAN.
2. **Flash**: command to flash (`esphome run esphome/texecom-bridge.yaml`). Note: first flash USB, subsequent flashes OTA.
3. **Boot observation**: tail logs (`esphome logs esphome/texecom-bridge.yaml`) and confirm "MQTT connected" appears.
4. **HA entity check**: open Home Assistant → Settings → Devices & Services → MQTT → Texecom Bridge. Within 30s of boot the three entities should appear:
   - `binary_sensor.texecom_bridge_status` — should read "Connected" (on)
   - `sensor.texecom_bridge_cpu_temperature` — should show a °C value
   - `sensor.texecom_bridge_wifi_signal` — should show a dBm value (negative number)
   Capture screenshots showing all three.
5. **LWT test**: power-cycle the Atom S3 (pull USB). Within 60s, `binary_sensor.texecom_bridge_status` should flip to "Disconnected" (off) and the two sensors should go `unavailable`. Reconnect USB; confirm all three recover within 30s.
6. **Broker-restart test**: on the host running Mosquitto, run `sudo systemctl restart mosquitto`. Entities should briefly go unavailable, then recover automatically.
7. **WiFi-blip test**: cause a brief WiFi outage (e.g. block the device's MAC at the AP for 30s, then unblock). Device should reconnect; entities should recover without restarting HA.
8. **Topic inspection** (optional): use `mosquitto_sub -h <broker> -t 'texecom-bridge/#' -v` to confirm topics like `texecom-bridge/status` carry the expected payloads (`online`/`offline`).

End the runbook with a "Phase close criteria" checklist mirroring the success criteria from ROADMAP. The user ticks each box and dates it once each test passes; that's the artefact that closes Phase 1.5.
