# Phase 1.5 — MQTT Bring-up: Validation Runbook

This runbook is the artefact that **closes Phase 1.5**. Plan 01.5-01 added an
`mqtt:` block plus three device-health entities to `esphome/texecom-bridge.yaml`.
The proof points (entities visible in HA, LWT works, broker/WiFi recovery)
require a real Mosquitto broker and a real Home Assistant instance, so they
are validated manually here. Run each step against your hardware in order.

---

## 0. Prerequisites

- Atom S3 hardware, USB cable, Phase 1 wiring still in place (panel can be
  disconnected for Phase 1.5 — no panel data is involved).
- Existing Mosquitto broker reachable on the LAN.
- Home Assistant with the MQTT integration already configured against that
  broker.
- ESPHome installed locally (`pip install esphome`) **or** via Docker.
- A way to subscribe to MQTT topics from a workstation (e.g. `mosquitto_sub`
  or MQTT Explorer) — used by step 8.

## 1. Pre-flight: secrets

Open `esphome/secrets.yaml` (gitignored, local-only) and confirm the three
new MQTT keys are present with **real** values. They were added as
placeholders to `secrets.yaml.example` by Plan 01.5-01:

```yaml
mqtt_broker_host: "192.168.x.y"        # IP or hostname of your Mosquitto box
mqtt_broker_username: "esphome"        # MQTT user; "" if anonymous-allowed
mqtt_broker_password: "your-password"  # ""  if anonymous-allowed
```

If the keys are missing, copy them across from `esphome/secrets.yaml.example`
and fill in real values. **Do not commit `secrets.yaml`.**

Sanity-check the broker is reachable from your dev box:

```
ping <mqtt_broker_host>
mosquitto_sub -h <mqtt_broker_host> -u <user> -P <pass> -t '$SYS/#' -C 1
```

## 2. Flash

First flash over USB (DFU/serial). Subsequent flashes can go OTA over WiFi.

```
esphome run esphome/texecom-bridge.yaml
```

Confirm compile succeeds and the firmware uploads without error.

## 3. Boot observation — logs

Tail logs and look for the MQTT bring-up sequence:

```
esphome logs esphome/texecom-bridge.yaml
```

Within ~15s of boot you should see:

- `[wifi]` association success
- `[mqtt] Connecting to MQTT...`
- `[mqtt] MQTT Connected!`
- Discovery payloads being published on `homeassistant/...` topics
- Birth message published: `texecom-bridge/status = online (retain)`

If `MQTT Connected!` does not appear, double-check broker host/username/
password in `secrets.yaml`, broker ACLs, and LAN reachability.

## 4. HA entity check

Open Home Assistant → **Settings** → **Devices & Services** → **MQTT** →
**Texecom Bridge**. Within 30s of boot the three entities should appear:

- `binary_sensor.texecom_bridge_status` — should read **Connected** (on)
- `sensor.texecom_bridge_cpu_temperature` — should show a °C value
  (typically 35–55 °C on an idle ESP32-S3)
- `sensor.texecom_bridge_wifi_signal` — should show a dBm value
  (negative number, e.g. -55 dBm)

Capture screenshots of the device card with all three entities visible and
their live values. Both telemetry sensors should refresh at most every 60s.

## 5. LWT test (device disconnect)

With HA showing all three entities healthy, **pull USB** (power-cycle the
Atom S3). Watch HA:

- Within ~60s, `binary_sensor.texecom_bridge_status` should flip to
  **Disconnected** (off). This is HA reacting to the retained
  `texecom-bridge/status = offline` LWT payload Mosquitto delivered when
  the device's TCP keepalive expired.
- The two `sensor.*` entities should go to state **unavailable**.

Reconnect USB. Within 30s of WiFi+MQTT reconnect, all three entities should
recover (status back to Connected, sensor values refresh).

## 6. Broker-restart test

On the host running Mosquitto:

```
sudo systemctl restart mosquitto
```

Watch HA: entities should briefly go **unavailable**, then recover
automatically once the broker is back. **No HA restart should be required.**
The device's MQTT client must reconnect on its own and re-publish the birth
message.

## 7. WiFi-blip test

Cause a brief WiFi outage to the Atom S3 — the simplest reproducible method
is to block its MAC address at the access point for ~30s, then unblock.
(Alternatively: power-cycle the AP, or `iw dev <iface> disconnect` if you
have CLI control.)

Expectations:

- During the outage: status entity flips off (after LWT timeout), sensors
  go unavailable.
- After WiFi returns: device reconnects, entities recover within ~30s.
- HA does **not** need to be restarted.

## 8. Topic inspection (optional but recommended)

From your workstation, dump everything under `texecom-bridge/`:

```
mosquitto_sub -h <broker> -u <user> -P <pass> -t 'texecom-bridge/#' -v
```

Confirm:

- `texecom-bridge/status` carries `online` (retained) while the device is up
- After step 5/6/7, `texecom-bridge/status` carries `offline` during the
  outage and flips back to `online` on recovery
- Discovery messages on `homeassistant/binary_sensor/...` and
  `homeassistant/sensor/...` reference the same `availability_topic` so HA's
  per-entity availability is wired through cleanly
