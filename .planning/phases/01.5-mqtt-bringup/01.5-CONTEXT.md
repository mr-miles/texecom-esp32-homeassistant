# Phase 1.5: MQTT Bring-up (Bridge Health to HA) -- Context

## Phase Goal
Prove the Atom S3 can register with Home Assistant via MQTT auto-discovery by publishing three built-in device-health sensors. No Texecom protocol decode involved — this de-risks the MQTT/discovery path before Phase 3 commits to it for panel data, and exercises the broker/network/HA integration end-to-end without needing the panel bench.

## Requirements Covered
- **REQ-010 — Bridge health telemetry**: Publish bridge online/offline status, CPU temperature, and WiFi signal strength to MQTT with HA auto-discovery, independent of any Texecom panel data.
- **REQ-005 (initial discovery scaffold)**: Establishes the MQTT topic/discovery scaffolding that Phase 3 will extend with decoded panel data. Phase 1.5 ships only the device-health half; panel data is layered on later.

## What Already Exists (from prior phases)
From Phase 1 (Plans 01-01 partial, 01-02 done):
- `esphome/texecom-bridge.yaml` — full ESPHome config with `wifi:`, `api:`, `ota:`, `web_server:`, `uart:`, `texecom:` blocks. **No `mqtt:` block yet.**
- `esphome/secrets.yaml.example` — placeholder file with `wifi_*`, `ota_password`, `api_encryption_key`. **No MQTT secrets yet.**
- `esphome/secrets.yaml` — local (gitignored) with real values for the keys above.
- `components/texecom/*` — the C++ component (TCP↔UART bridge + capture). **No MQTT publishing logic.**
- CI green via `.github/workflows/ci.yml`; host-side tests in `tests/`.

What is **explicitly out of scope** for Phase 1.5:
- Any C++ code in `components/texecom/`. This phase is pure ESPHome YAML using built-ins.
- Decoded panel data (zones, areas, etc.) — that's Phase 3, layered on top of the discovery scaffold built here.
- Wintex-mode exclusivity logic for MQTT pause/resume — Phase 3 problem; bridge health publishes regardless of bridge mode (it's device telemetry, not panel data).

## Key Design Decisions

- **Single plan, single wave.** Tiny slice (3 sensors, all ESPHome built-ins). Splitting across waves adds more bookkeeping than it's worth.
- **No new C++.** All three sensors are ESPHome built-ins:
  - Bridge online/offline → `mqtt.birth_message` + `mqtt.will_message` on a status topic, plus a HA discovery `binary_sensor.connectivity` linked to the same topic.
  - CPU temperature → `sensor.internal_temperature` (ESP32 platform built-in, ESP32-S3 supports it).
  - WiFi signal → `sensor.wifi_signal` (built-in).
- **Independent of Texecom code.** This slice changes only `esphome/*.yaml` files. The `components/texecom/` C++ code is untouched, so there's zero regression risk to the Phase 1 bridge work.
- **MQTT broker creds via secrets.** Add `mqtt_broker_host`, `mqtt_broker_username`, `mqtt_broker_password` to `secrets.yaml.example`; the user fills in `secrets.yaml` from their existing Mosquitto credentials.
- **Senior developer leads.** ROADMAP recommends senior-developer + api-tester + evidence-collector. The api-tester and evidence-collector roles are post-flash manual user steps (HA UI screenshots, MQTT topic inspection); they don't run in `/legion:build`. Senior-developer handles the YAML and the validation runbook. The user runs the runbook themselves once hardware is ready — captured as an explicit hand-off, mirroring the Plan 01-03 pattern.
- **No unit tests required.** Per PROJECT.md, "Unit tests required for all non-trivial code." This slice is YAML config of built-ins — no non-trivial logic to test. The validation runbook substitutes for an automated test where automation isn't appropriate.
- **Validation deferred to a runbook.** Like Plan 01-03 (Wintex validation), the actual proof points (entities visible in HA, broker-restart recovery, WiFi-blip recovery) need a real broker + real HA. Documented in `01.5-VALIDATION.md` as a step-by-step the user runs.
