## 1. Final shape of the `mqtt:` block

Placed in `esphome/texecom-bridge.yaml` between `api:` and `ota:`, with a
phase-1.5 explanatory comment block above it.

```yaml
mqtt:
  broker: !secret mqtt_broker_host
  username: !secret mqtt_broker_username
  password: !secret mqtt_broker_password
  topic_prefix: texecom-bridge
  birth_message:
    topic: texecom-bridge/status
    payload: online
    retain: true
  will_message:
    topic: texecom-bridge/status
    payload: offline
    retain: true
```

- **broker / username / password**: all `!secret`-resolved
  (`mqtt_broker_host`, `mqtt_broker_username`, `mqtt_broker_password`).
- **topic_prefix**: `texecom-bridge` (matches device name; keeps Phase 3
  panel data under the same namespace).
- **birth payload**: `online` on `texecom-bridge/status`, retained.
- **will payload**: `offline` on `texecom-bridge/status`, retained.
- **discovery**: enabled by default in ESPHome — no explicit flag needed;
  HA auto-creates entities under the `homeassistant/...` discovery prefix.

## 3. Connectivity entity — approach used

The plan specified `binary_sensor.template` with a `lambda: |- return true;`
body and a `device_class: connectivity`, relying on ESPHome's auto-published
MQTT availability topic (driven by the LWT/birth messages on
`texecom-bridge/status`) to flip HA's per-entity availability state.

**That is exactly what was implemented — no deviation.**

Note for a possible future tidy-up (out of scope here, flagged for the
record): the `lambda: return true;` on the connectivity entity is
near-redundant — ESPHome's auto-published availability topic already drives
the unavailable→available transition for every entity tied to this device,
and a dedicated `binary_sensor.connectivity` platform pointed at the LWT
topic would express intent more directly. The plan body explicitly told us
not to change approach here, and the extra entity is harmless. If
review/refactor surfaces this later, the swap is a one-block YAML change.
